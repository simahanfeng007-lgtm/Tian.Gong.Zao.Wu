一、阶段定位

L6.33 只处理执行链验证底座：分片回归、独立超时、进程收割、证据落盘与汇总恢复。
本轮没有补接 Provider / Skill / Handoff / 情志 / 前端 / 安装包等其他系统，避免把执行主链冻结和系统接入混在一起。

二、核心改动

1. 新增 tiangong_agent_runtime/test_shard_runner.py。
   - TestShardSpec / TestShardResult / TestShardRunSummary。
   - build_pytest_shard(...)。
   - run_test_shards(...)。
   - write_shard_summary(...)。
   - collect_existing_shard_results(...)。
   - 每个 shard 独立 stdout / stderr / result.json / rerun_command。

2. 新增 timeout reaper。
   - 每片独立 timeout_seconds。
   - timeout 独立标记为 timeout，不冒充 passed 或 failed。
   - 超时后尽力收割进程组。
   - result.json 已落盘但 worker 仍不退出时，也允许父调度器收割尾部 worker。

3. 新增汇总恢复能力。
   - 如果外层聚合命令被环境超时器打断，但各 shard 已经写出 result.json，可通过 collect_existing_shard_results(...) 重建 shard_summary.json / shard_report.txt。
   - 这是针对当前环境中“pytest summary 已打印、result.json 已写出，但进程/管道尾部不回收”的工程化治理。

4. 新增 tests/test_l6_33_test_shard_runner.py。
   - 覆盖通过 shard。
   - 覆盖 timeout shard。
   - 覆盖 optional failure → partial。
   - 覆盖 pytest argv 构造不走 shell string。
   - 覆盖边界 flag。
   - 覆盖从既有 result.json 重建汇总。

四、重要说明

本轮仍不声明完整全量 pytest 通过。
原因：当前环境仍会复现若干测试进程/子进程尾部不回收问题。L6.33 的目的正是把这类问题分片化、证据化、可收割化，而不是把一次大聚合 pytest 卡死误报为失败或通过。

本轮已经做到：
- 单个 shard 独立产物落盘。
- timeout 明确标记。
- result.json 已落盘时可以重建汇总。
- 不因单一 shard 破坏全部验证证据。
