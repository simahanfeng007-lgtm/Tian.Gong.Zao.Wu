2. L0-L2 继承关系
============================================================

2.1 L0：零依赖原语层

L0 是 L3 的最底层事实语言来源。L3 只能复用 L0 的身份、引用、时间、结果、错误、序列化、追踪、事件、观察、信号、指标、内容、消息、行动意图、计划引用、风险事实、租约、状态引用、生命周期、失败、事务、删除、审计、证据、版本、命名空间、关系、验证等原语。

L3 不得让 L0 产生上层业务含义，不得向 L0 添加编排器、调度器、执行器、模型客户端、工具客户端、状态仓库或插件宿主。

2.2 L1：端口协议层

L1 是 L3 与上下层交互的协议来源。L3 应通过 L1 已定义的边界说明、请求/响应信封、PortResult/CoreResult、Skill、ToolGroup、ToolRelease、Model、Observation、Memory、Retrieval、Learning、Validation、Change、Experiment、SecurityBoundary、StateContinuity、ActionEffect 等端口族表达“请求”。

L3 不实现 L1 端口背后的真实外部能力，不替代端口实现者，不把协议对象变成执行对象。

2.3 L2：状态层

L2 是 L3 的状态输入与状态建议输出目标。L3 可以读取或引用 L2 状态切片、L2 投影、L2 状态引用和第九阶段新增数学/情感/动态驱动状态，但不能修改 L2 对象，不能持久化 L2 状态，不能把 L2 状态变成执行器。

L3 输出的状态变化只能表现为 StateTransitionAdvice、StateUpdateSuggestion、OrchestrationProjection、OrchestrationResult 等“建议对象”或“结果摘要对象”。真正记录仍由 L2 后续状态写入链处理，真正执行仍由 L4 处理，真正边界裁决仍由 L5 处理，真实子系统服务仍由 L6 处理。

============================================================
3. L3 一句话定义
============================================================

L3 是“流程如何推进”的编排层：它把 L2 状态、L1 协议请求、大模型主控、Skill 直显、工具组释放、模型意图、边界请求、执行请求、观察回流、候选验证与恢复流程组织成可续接、可审计、可恢复、可解释的编排计划；但它不执行动作、不调用模型、不调用工具、不做权限裁决、不实现存储、不实现 L6 子系统。

4. L3 职责范围
============================================================

L3 可以做：

1. 接收 L2 状态切片和 L1 请求信封。
2. 组织 run / task / turn / step 的流程连续性。
3. 组织 Skill 直显、Skill 选择、ToolGroup 释放、ToolIntent、BoundaryCheck、ExecutionRequest、ObservationResult、Feedback、Candidate、Validation、Recovery 的顺序。
4. 输出 OrchestrationPlan、OrchestrationStep、OrchestrationTransition、OrchestrationGraph、OrchestrationResult、OrchestrationProjection。
5. 根据 L2 状态判断流程是否缺状态、断链、等待、失败、取消、需恢复或需重试。
6. 构造发往 L5 的边界审查请求对象，但不裁决。
7. 构造发往 L4 的执行派发请求对象，但不执行。
8. 构造发往 L6 的子系统服务请求对象，但不实现子系统逻辑。
9. 计算或组织可解释数学评分、路径排序、状态转移建议，但这些输出只能作为建议。
10. 接入 L2 第九阶段的数学状态、情感状态、动态驱动状态，形成 AffectiveWeightInput、DynamicDriveInput 和 MathEvaluation/MathRecommendation。
11. 为大模型执行力提供结构化“下一步应该请求什么”的流程支架。
12. 为工程生命体提供可审计、可恢复、可演化的运行骨架。

6. L3 与 L0/L1/L2 的边界
============================================================

6.1 对 L0 的边界

L3 只复用 L0 的原语和稳定序列化能力，不修改 L0，不引入上层名词到 L0，不在 L0 放置编排对象。

6.2 对 L1 的边界

L3 使用 L1 端口协议表达请求，不实现端口。L3 的请求对象应能够被未来 L4/L5/L6 适配为真实请求，但 L3 本身不连接任何外部资源。

6.3 对 L2 的边界

L3 读取 L2 状态引用和状态切片，不改写 L2；L3 输出状态更新建议，不持久化状态。L2 的 frozen + slots、稳定序列化、稳定 hash 口径必须继续保持。

6.4 L3 的公共对象边界

L3 的公共请求、结果、计划、步骤、转移、数学输入/输出对象默认使用 dataclass(frozen=True, slots=True)。如 builder/planner 内部确需可变结构，只能在局部纯内存构造期使用，不得暴露为公共状态或公共请求对象。

7. L3 与 L4/L5/L6 的边界
============================================================

7.1 L4 执行层预留

未来 L4 职责：接收 L3 生成的执行请求，在 L5 允许后调用真实模型适配器、工具适配器、文件/网络/终端/桌面等执行适配器，并返回执行结果、观察、错误和审计引用。

L3 对 L4 只预留：ExecutionRequest、ExecutionPlanRef、ExecutionStepRef、ExecutionTokenRef、ExecutionResultRef、ExecutionFailureRef、ExecutionResumeRef、ExecutionCancelRef。

L3 不实现执行器，不持有真实工具对象，不持有模型客户端，不调用适配器。

7.2 L5 边界层预留

未来 L5 职责：权限裁决、风险评分、资源限额、安全边界、确认票据、凭据治理、租约治理、审计门禁、允许/拒绝/降级/要求确认。

L3 对 L5 只预留：BoundaryCheckRequest、RiskReviewRequest、PermissionReviewRequest、ConfirmationRequest、LeaseRequest、PolicyDecisionRef、BoundaryDecisionRef、DenialReasonRef。

L3 不实现策略引擎，不给出最终许可，不覆盖人工确认，不读取凭据，不发放租约。

7.3 L6 子系统层预留

未来 L6 职责：记忆系统、Skill 系统、Tool 注册与工具组系统、检索系统、学习系统、情感系统、自我迭代、自我进化、观察系统、审计系统、UI/前端/连接器适配和其他插件化子系统。

L3 对 L6 只预留：SubsystemRequest、SkillCatalogRequest、SkillResolveRequest、ToolGroupResolveRequest、MemoryServiceRequest、RetrievalServiceRequest、LearningServiceRequest、AffectiveServiceRequest、EvolutionServiceRequest、ObservationServiceRequest、AuditServiceRequest。

L3 不实现记忆算法、检索算法、学习算法、情感算法、进化算法、观察器或 UI。

8. L3 对新版主链的支撑方式
============================================================

新版主链：

大模型 → Skill 直显 → Skill 选中 → 工具组释放 → 模型提出工具调用意图 → L5 边界裁决 → L4 执行 → 观察结果 → 反馈 → 候选 / 验证 / 回滚状态 → L2 记录

L3 的位置是“流程支架”。它不替代大模型主控，不替代 L5 边界，不替代 L4 执行，不替代 L6 专业子系统。

L3 的支撑方式：

1. 通过 L2 的 ModelVisibleStateProjection、L3HandoffProjection、RuntimeSliceProjectionState 等投影引用组织当前模型可见流程切片。
2. 将 Skill 直显视为一个编排步骤，而不是执行授权。
3. 将 ToolGroup 释放视为一个请求编排步骤，而不释放真实工具。
4. 将 ToolIntent 视为模型提出的意图状态或请求输入，而不是执行动作。
5. 将 BoundaryCheck 编排为 L5 请求，不在 L3 做放行。
6. 将 ExecutionRequest 编排为 L4 请求，不在 L3 执行。
7. 将 ObservationResult 编排为回流输入，不在 L3 采样观察。
8. 将 Feedback、Candidate、Validation、Recovery 编排为后续建议和请求，不生成真实补丁、不执行验证、不执行回滚。

============================================================

1. L3 输入应使用 L2StateIdentity、TypedRef、StateSnapshotRef、ProjectionFragmentState、L3HandoffProjection、RuntimeSliceProjectionState 等引用或不可变状态对象。

5. L3 的建议可由未来 L2 状态写入链记录；L3 自己不持久化。
6. L3 的结果必须可稳定序列化、可 hash、可 diff、可快照测试。

============================================================
10. L3 如何组织 run / task / turn / step
============================================================

L3 应按四层连续性组织：

1. Run：一次完整运行会话或生命活动片段，包含 run_id、入口、目标、边界摘要、当前状态切片、连续性状态和恢复点引用。
2. Task：Run 内的任务单元，包含 task_id、目标向量、约束集合、上下文引用、候选路径和状态转移建议。
3. Turn：模型交互回合，包含输入信封、模型可见投影、模型反馈引用、意图引用和观察回流引用。
4. Step：最小编排步骤，包含 step_id、step_kind、输入引用、输出建议、前置条件、后置条件、边界要求、失败路径和续接提示。

第二阶段才实现 Run/Task/Turn/Step 的专门编排对象和连续性评分。第一阶段只建立基础 OrchestrationStep、OrchestrationPlan、OrchestrationTransition、OrchestrationContext，不进入真实 run/task 调度。

============================================================
11. L3 全系统数学驱动设计
============================================================

11.1 总原则

固定规则只负责不可突破的架构宪法和安全底线；底线内的动态判断尽量转化为状态特征、目标向量、约束集合、评分向量、路径排序和状态转移建议。

标准链路：

输入状态 → 提取特征 → 目标向量 → 约束集合 → 评分向量 → 路径排序 → 编排建议 → L5 边界校验 → L4 执行 → L2 记录结果

11.2 L3 数学模型对象族

L3 至少应规划下列数学对象方向：

1. MathFeatureVector：从 L2 状态切片提取的特征集合。
2. MathObjectiveVector：当前任务目标、系统目标、用户目标和工程生命体目标的向量化表达。
3. MathConstraintSet：硬约束、软约束、资源约束、上下文约束、工具暴露约束、确认需求等集合。
4. MathScoreVector：各候选路径的多维评分。
5. MathEvaluation：某个候选路径或状态转移的评估结果。
6. MathRecommendation：基于评分和约束的建议，不是执行令。
7. TransitionScore：状态转移建议的评分。
8. RouteRanking：候选路径排序。
9. StabilityIndex：稳定性指标。
10. ContinuityIndex：连续性指标。
11. ReversibilityIndex：可逆性指标。
12. ExecutionReadinessScore：执行准备度评分。
13. LearningValueScore：学习价值评分。
14. AffectiveWeightInput：情感/情志权重输入。
15. DynamicDriveInput：动态驱动权重输入。
16. StateTransitionAdvice：状态转移建议。
17. OrchestrationRanking：编排路径排序。

11.3 评分方向

可采用以下策划公式方向：

ActionScore =
目标匹配度
+ 成功概率
+ 信息增益
+ 学习价值
+ 连续性收益
+ 稳定性收益
+ 用户偏好匹配
+ 可逆性收益
- 风险代价
- 资源成本
- 上下文成本
- 工具暴露成本
- 不可逆代价
- 失败重复惩罚

注意：第一阶段不实现复杂算法。第一阶段只定义通用数据结构、纯函数接口或纯评估对象，最多允许做确定性、可解释、无外部依赖的基础组合函数，不做 Skill 评分、工具评分、风险评分、学习评分或进化评分复杂策略。

11.4 数学模型边界

1. 数学模型只给建议，不给执行令。
2. 数学模型不代替 L5 权限/风险/确认/租约裁决。
3. 数学模型不代替 L4 执行。
4. 数学模型不代替 L6 记忆、检索、学习、情感、进化专业算法。
5. 数学模型输出必须可解释、可序列化、可测试。
6. 数学模型不得覆盖架构宪法和安全底线。

12. L3 情感/情志权重接入设计
============================================================

12.1 继承口径

稳定情感底色 + 临时情感 / 欲望 → 情感总色彩。

七情影响说话方式、表达强度、交流风格。六欲影响做事倾向、主动性、探索/稳定/学习/修复等偏好。

12.2 L3 接入方式

L3 不实现情感算法，只读取或引用 L2 的情感状态：EmotionBaseState、EmotionTransientState、DesireTendencyState、ExpressionBiasState、ActionBiasState、AffectiveColorState、AffectiveBoundaryState。

L3 将其转化为 AffectiveWeightInput，作为数学编排模型的权重输入，例如：

1. curiosity_weight
2. caution_weight
3. persistence_weight
4. repair_weight
5. learning_weight
6. exploration_weight
7. stability_weight
8. confirmation_preference_weight
9. social_warmth_weight
10. directness_weight

12.3 可影响范围

情感/情志权重可以影响：

1. 候选路径排序倾向。
2. 探索/谨慎/修复/学习/稳定偏好。
3. 是否更倾向请求确认。
4. 是否更倾向继续尝试或切换路径。
5. 表达更直接或更温和。
6. 编排建议的解释文本语气。

12.4 禁止影响范围

情感/情志权重不得影响：

1. 是否绕过 L5 边界。
2. 是否直接执行。
3. 是否读取敏感数据。
4. 是否删除、覆盖或迁移文件。
5. 是否调用高危工具。
6. 是否提升权限。
7. 是否自动合入。
8. 是否跳过验证。
9. 是否覆盖用户显式要求。
10. 是否覆盖架构宪法。

13. L3 动态驱动状态接入设计
============================================================

L3 应读取或引用 L2 第九阶段 DynamicWeightState、SystemDriveState、PreferenceWeightState、StabilityPressureState、RiskPressureState、ExplorationPressureState、ExecutionReadinessState、DynamicDriveEvaluationRefState。

这些状态在 L3 中转为 DynamicDriveInput，用于影响路径排序、优先级、恢复建议、验证建议和执行准备度建议。

动态驱动只允许表达：

1. 当前系统更偏探索还是利用。
2. 更偏谨慎还是速度。
3. 更偏稳定还是创新。
4. 更偏学习还是直接完成任务。
5. 更偏修复、恢复、验证还是继续推进。
6. 当前执行准备度是否不足，应先补状态、补确认、补验证或降级路径。

动态驱动不得产生执行令，不得选择 Skill，不得释放工具，不得裁决权限，不得写状态存储。

14. L3 核心对象族
============================================================

14.1 编排基础对象

1. OrchestrationRequest：一次编排请求的入口对象，记录请求来源、目标、输入状态引用、约束和输出期望。
2. OrchestrationContext：编排上下文，记录 trace、scope、当前 run/task/turn/step 引用、状态投影引用、数学输入引用和边界提示。
3. OrchestrationStep：单个编排步骤，记录步骤类型、输入引用、输出建议、前置条件、后置条件、失败路径和下一步候选。
4. OrchestrationPlan：步骤集合和顺序关系，不是执行计划。
5. OrchestrationGraph：步骤之间的有向关系、分支、等待、恢复和降级路径。
6. OrchestrationTransition：状态转移建议和流程转移边。
7. OrchestrationInvariant：编排层不变量，例如“不得执行”“不得裁决”“不得持久化”。
8. OrchestrationProjection：面向模型、人类、调试、审计、L4/L5/L6 的编排摘要。
9. OrchestrationResult：编排输出，包含建议、排序、请求对象引用和未满足条件。
10. OrchestrationError：编排错误事实，不抛出业务异常。

14.2 后续阶段预留请求对象

1. BoundaryReviewRequest：第五阶段开始定义，发往 L5，不裁决。
2. ExecutionDispatchRequest：第五阶段开始定义，发往 L4，不执行。
3. SubsystemServiceRequest：第六阶段开始定义，发往 L6，不实现服务。

第一阶段不应实现上述后续请求对象的完整版本。

18. L3 风险清单与控制措施
============================================================

风险 1：把 L3 写成旧 Runtime 或“神枢”。
控制：禁止真实运行循环、工具执行、模型调用、全局调度器；L3 只生成编排计划和请求对象。

风险 2：数学模型越权成为裁决器。
控制：所有 MathEvaluation/MathRecommendation 必须包含 recommendation_only 或 equivalent 边界语义；测试中扫描不得出现 grant/allow/execute 等放行语义作为最终裁决。

风险 3：情感权重越权。
控制：AffectiveWeightInput 只参与排序和解释；不得生成 ExecutionRequest、BoundaryDecision 或权限提升字段。

风险 4：第一阶段过度开发。
控制：第一阶段禁止 run_orchestrator、skill_orchestrator、boundary_orchestrator、execution_dispatcher、subsystem_router、risk_engine、learning_engine。

风险 5：L3 污染 L2。
控制：L3 只 import L2 公共状态对象和引用，不修改 L2 源码；测试确认 L2 文件 hash 或 git diff 未变。

风险 6：隐藏 IO 或网络访问。
控制：静态扫描 open、Path.write_text、requests、urllib、socket、subprocess、os.system、sqlite、db、client 等模式。

风险 7：对象可变性破坏可审计性。
控制：公共数据对象必须 frozen + slots；list/dict 默认值不得暴露为可变默认。

风险 8：L4/L5/L6 接口提前实装。
控制：L3 只定义请求/引用/建议，不实现适配器、策略引擎、子系统。
