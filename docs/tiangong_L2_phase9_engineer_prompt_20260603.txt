二、本阶段背景
────────────────────────────────

L2 第一至第八阶段已经完成 L2 状态层主干：

1. 基础状态骨架；
2. 生命体运行状态与连续性；
3. Skill / ToolGroup / Model / Action 状态；
4. 控制面、资源、环境、安全状态；
5. 观察面状态；
6. 记忆、上下文、检索、学习状态；
7. 候选、变更、迭代、进化、实验、验证、恢复状态；
8. 组件、兼容迁移、状态投影与 L2 总收口。

现在用户新增全局方向：

“所有系统都应由数学模型动态驱动。固定判定只保留不可突破的架构宪法和安全底线；其他行为应尽量转化为状态特征、目标函数、约束集合、评分向量和状态转移建议。”

同时，旧版天工造物已有情感系统概念：

稳定情感底色 + 临时情感 / 欲望 → 情感总色彩

七情影响说话方式和表达倾向。
六欲影响做事倾向和主动性。
情志层只给表达/倾向，不给执行令，不能绕过边界和执行链。

因此，L2 需要补充三类状态入口：

1. 数学模型状态入口；
2. 情感 / 情志状态入口；
3. 全系统动态驱动权重状态入口。

注意：
L2 只记录这些状态，不计算它们。
数学计算留给 L3。
情感系统真实算法留给 L6。
最终边界裁决留给 L5。
真实执行留给 L4。

────────────────────────────────
三、本阶段定位
────────────────────────────────

L2 第九阶段只定义状态事实对象。

本阶段只回答：

1. 后续数学模型使用了哪些特征？
2. 当前目标向量是什么？
3. 当前约束集合是什么？
4. 数学评分结果如何被记录？
5. 数学建议如何被记录？
6. 情感底色、临时情感、欲望倾向、情感总色彩如何被记录？
7. 情感如何影响表达倾向和行为倾向？
8. 全系统动态驱动权重如何被记录？
9. 探索、稳定、谨慎、学习、恢复、执行准备度等压力如何被记录？
10. 这些状态如何被 L3 引用、被 L5 审查、被 L6 子系统实现？

本阶段不回答：

1. 数学模型如何计算；
2. 情感系统如何生成情感；
3. 欲望倾向如何学习；
4. 哪个动作最终执行；
5. 哪个风险最终放行；
6. 哪个工具最终调用；
7. 哪个 Skill 最终选择；
8. 哪个状态最终持久化；
9. 哪个候选最终合入。

一句话：

L2 第九阶段 = 数学模型、情感系统、动态驱动机制的状态预留层。
它只记录状态，不做计算，不做执行，不做裁决。

────────────────────────────────
四、必须坚持的分层边界
────────────────────────────────

L2：
- 记录状态；
- 记录特征；
- 记录目标；
- 记录约束；
- 记录评分结果；
- 记录情感状态；
- 记录动态权重；
- 记录引用；
- 记录置信度；
- 记录边界状态。

L3：
- 读取 L2 状态切片；
- 计算数学评分；
- 进行流程编排；
- 输出状态转移建议；
- 输出边界请求；
- 输出执行请求；
- 不执行工具；
- 不裁决权限。

L4：
- 接收 L3 执行请求；
- 在 L5 允许后执行真实动作；
- 调用模型、工具、文件、网络、终端等真实适配器；
- 返回结果和观察。

L5：
- 接收 L3 边界请求；
- 做权限、风险、确认、租约、资源、安全、审计裁决；
- 不被情感或数学模型绕过。

L6：
- 实现真实子系统；
- 包括情感系统、记忆系统、Skill 系统、ToolGroup 系统、检索系统、学习系统、进化系统等；
- 可以各自拥有专用数学模型；
- 但不能绕过 L3/L5/L4 主链。

硬边界：

数学模型只能提供建议。
情感系统只能影响表达和倾向。
动态权重只能影响排序和优先级。
任何执行都必须经 L5 边界与 L4 执行层。
L2 不允许产生执行令。

────────────────────────────────
九、数据结构硬要求
────────────────────────────────

所有新增状态对象必须满足：

1. 使用 @dataclass(frozen=True, slots=True)。
2. 字段必须可稳定序列化。
3. 字段默认值必须安全，不使用可变默认值。
4. 不持有真实文件句柄、数据库连接、网络连接、模型客户端、工具对象、回调函数、协程、线程、锁或运行时执行器。
5. 只记录 id、ref、status、kind、scope、timestamp、summary、hash、score、weight、confidence、boundary_status 等状态事实。
6. 数值字段必须明确语义，建议 0.0-1.0 归一化字段使用 normalized/value/weight 命名。
7. 长文本必须克制，优先 summary、hash、ref。
8. 必须能参与快照、增量、稳定 hash、等值比较。
9. 必须有中文 docstring。
10. 禁止 import L3、L4、L5、L6 或任何未来层实现。
11. 禁止第三方依赖。
12. 禁止任何真实 IO、模型调用、工具调用或外部副作用。

建议字段类型优先使用：

- str
- int
- float
- bool
- tuple
- frozenset
- enum
- datetime 字符串或 L0 时间类型
- L0/L1/L2 已定义的引用对象
- 本阶段状态对象

避免使用：

- dict 的可变嵌套结构；
- list；
- set；
- Any；
- Callable；
- object；
- 真实外部对象。

如果需要表达键值权重，建议使用稳定 tuple：

(("curiosity", 0.7), ("caution", 0.4))

不要使用可变 dict 默认值。
