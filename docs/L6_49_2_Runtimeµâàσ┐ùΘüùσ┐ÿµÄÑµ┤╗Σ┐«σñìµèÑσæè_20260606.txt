天工造物 / 临渊者 L6.49.2 Runtime 情志遗忘接活修复报告
============================================================

一、本轮结论

用户质疑成立：L6.49.1 验证了情感/遗忘引擎自身数值波动，但没有证明 Runtime 把两套引擎接活。
经源码复核，当前包内存在 P1 级接入缺陷：

1. 情志系统：RuntimeEntry 未正式持有 AffectiveStateEngine 的 previous_state，也未在任务发布时把情志 route 注入 Planner hint。
   结果是测试层可证明 engine 动态，但 Runtime 任务路径没有真实连续情志状态。

2. 遗忘系统：ForgetReviewRouter.review(record, vector) 只有单条记录复核签名；Runtime 没有批量生成 ForgettingScoreVector 并调用 review(record, vector) 的接线。
   结果是遗忘审查在 Runtime 任务路径没有真实运行。

二、本轮修复

1. RuntimeEntry 增加 Runtime 内部情志状态：
   - self._affective_engine
   - self._affective_router
   - self._affective_state
   - self._affective_route
   - self._last_affective_update_at
   - self._affective_turn_count

2. run_text / run_planner_execution_task 构造 planner context 前，调用 _build_live_cognitive_context_hint()：
   - derive emotion/desire sources
   - 调用 AffectiveStateEngine.evolve(..., previous_state=self._affective_state, elapsed_seconds=elapsed)
   - 更新 Runtime 内部 previous_state
   - 生成 AffectiveExecutionRoute planner hint

3. RuntimeEntry 增加 ForgetReviewRouter 接线：
   - 可注入官方 MemoryStoreBridge
   - 遍历 active_records()
   - 为每条 MemoryRecord 生成 ForgettingScoreVector
   - 调用 ForgetReviewRouter.review(record, vector)
   - 不物理删除、不改 memory store，只输出 review decision

4. 禁止静默吞错：
   - 遗忘审查失败不再 except pass
   - 错误进入 _last_forget_review_error，并重新抛出，方便测试和修理定位

5. 新增 Runtime 快照：
   - affective_runtime_snapshot()
   - forgetting_review_runtime_snapshot()

五、边界确认

1. 情志仍然只是 Planner hint，不授权、不拒绝、不派发工具、不覆盖质量门。
2. 遗忘仍然只是 review decision，不直接删除、不直接写 store、不物理瘦身。
3. 本轮不修改 ExecutionSpine / QualityGate / Audit / Rollback。
4. 本轮不绕过 Runtime，不新增 Provider 接入，不接触 API Key。

六、剩余风险

1. 当前 Runtime 只在 MemoryStoreBridge 注入后运行遗忘复核；若生产入口未提供 MemoryStoreBridge，则遗忘 review_count 会为 0，但不会报错。
2. 情志输入源目前由用户消息关键词和任务态启发式映射生成；后续可接真实压力场/审计失败/预算压力，但本轮不扩展模型。
3. full pytest 未跑完，不能声明全量通过。
